Building the circuit:

Below is a schematic of the circuit that needs to be built. In all cases but the stepper motor, the red wires are power, the black is ground, the yellow is analog and digital connections, and the purple are simple connectors. Note that although the signal pin of the analog light sensor is connected to pin 14, this pin is also an analog pin (A8) on the ESP 32 Feather (which is a different, but very similar, board than in the schematic). Just make sure that the sensor is connected to an analog-enabled pin. Additionally, the LED is representative of the laser which needs to be taken apart to be used in the circuit (this is why the alligator clips are needed). You can see from the pictures that the LED requires a MOSFET to be told when to turn on and off by the microcontroller. And finally, the capacitor is to prevent any unwanted voltage spikes.

MIDI:

The whole goal behind the laser harp is to break a beam of light, have that light hit the sensor which in turn tells the microcontroller to send a MIDI note that corresponds to the beam being broken. Therefore, before continuing I needed a MIDI for the microcontroller. It is easy to hook one up using your computer (not I am on windows). You will need this hairless MIDI to serial bridge. This allows for serial outputs from the microcontroller to become MIDI signals. To play these signals you can use VirtualMIDISynth. However, there are things called SoundFonts. These can be thought of as... well fonts for sound. Essentially, depending on the input, you can change the instrument that is being played. That said, you must download and use a SoundFont with only one instrument, or else you cannot control the note and the instrument. This was one of those things that took me a little bit to figure out and when we hit the code, you will see why this is necessary. You can download a simple piano SoundFont from here. I'm using the "piano" SoundFont that is on the page. After putting this SoundFont into VirtualMIDISynth and changing your MIDI out to VirtualMIDISynth you are essentially all set to play MIDI data. The remaining steps will be discussed in the code section.

The Code:

In our code we will use two libraries. The first is Accelstepper, a library full of secrets with documentation located here which can be downloaded from the Arduino IDE.

The second is the Arduino MIDI Library whose documentation can be found here and which can also be downloaded from the Arduino IDE.

At this point, we have all the parts to start our code. I will break it up into 'before the "setup" function' and 'in the "setup" and "loop" functions.' First, let us take a look at what is before the setup function

In this code we have the initialization of the MIDI communication, initialization of the stepper motor, the class which contains the "update" function telling a note to play or turn off, and the initialization of each beam as a note. A few comments: First, I imagine this code is not optimized. Second, you can find the note numbers here. Also, I declared each beam to send notes through a different channel, this should work fine but might be unnecessary.

Since the MIDI functionally "sendNoteOn/Off" is introduced in this block of code it is a good time to quickly look at what those functions do and how basic messages look like when they are received by the hairless serial to MIDI bridge. Below is a video of code running that plays an incrementing note every 1 second then shuts it off. It is mentioned in the next section the importance of the "Serial.begin(XXX);" line of code. Also, note that I do not show the VirtualMIDISynth in this video but I have it open with my wanted SoundFont chosen.

Now, let's look at what is inside the "Setup" and "Loop" functions.

Here in the "setup" function everything should be pretty straightforward. The only thing to note is that for the hairless serial to MIDI bridge to work it needs to be on the same baud rate. You can change the baud rate by going to "File" --> "Preferences" within the hairless MIDI bridge.

Within the loop function the first part is supposed to read the light level of the room. However, I found out last minute it seems as if the "delay(10);" line of code blocks the laser from being on which affects the reading. To fix this, removing this line of code should be sufficient. Within this part, it also sets the threshold for what the sensor must read to play a beam.

The next part is what runs the stepper motor. First, huge shoutout to Nathan for the immense amount of help he gave me working with the very elusive AccelStepper library. By incrementing "n" we can increase the position of the stepper by a defined amount of steps. After a defined amount of increments, "dir" reverses sign reversing the direction teh stepper moves. I found that to work properly, the "setSpeed" line of code must be negative when turning backward, which is not exactly obvious since speed is a scalar. Also, the condition around "stepper.runSpeed();" is necessary, or else the stepper will turn continuously. I believed I narrowed why this is down to that without the condition, the stepper will hit "distanceToGo() == 0" then hit the line "stepper.runSpeed();" before the block of code that makes the stepper increment position, making the stepper move when it shouldn't yet. Additionally, within this section, it is important to note that "waitTime" must be declared larger than the time it takes to loop through the code (approx 30+ millis). I believe this has something to do with when the conditions "(stepper.distanceToGo() == 0)" and "(currentMillis - prevMillis >= waitTime)" are executed although I am not sure. Finally, the "Update" functions are called for each beam to see if the conditions to play a note are satisfied.

More Comments/Improvements/Future Plans:

Before moving on to my 3D printing process, I quickly want to talk about potential improvements, directions to go, and comments about the code. The biggest thing is, when I was running my tests I noticed that it appears as if the "sendNoteOn" command is blocking and the laser is weaker than it should be. I'm grouping these because I think they are closely related. It appears that when the beam is broken and the light sensor picks up on it, the stepper starts to slow down, skip steps, and move a little more erratically while, at the same time, the laser gets brighter. I believe what is happening is that the stepper motor driver is calling less power since it's being blocked allowing for more current to go to the laser. One workaround to the laser's strength would be to hook it up to external batteries or using the 5V pin on the microcontroller (which the ESP32 Feather does not have). The workaround to it being blocking is to just make the whole code blocking. I was very focused on not making everything blocking, but I think that if the motor turns, check to see if a note should be played, then stop if yes or continue if no would address most of the problems I have encountered. This is something I am going to look into to see if I can get this to work.

One other thing I could add which I incorporated space into my 3D print for is an end effector for the motor. I believe having a hull sensor with a magnet attached to the base of the mirror would allow to motor to be set into place without manually turning it (which is what I currently have to do).

Also, there I noticed a library that is called "ESP32-BLE-MIDI"" that I didn't see earlier. After a quick look, I think that this might be effective in getting rid of the need for the hairless serial to MIDI bridge and would allow the whole circuit to be more independent from the computer. I am not sure what will receive the MIDI signal on the computer, but it is still something I am interested in looking into.

The 3D Print:

I am extremely happy with how the 3D print turned out. I planned to make something that will hold all the parts effectively, deal with unnecessary wires, while also being fairly compact. The idea was to create a long breadboard holder, on which I could mount the driver, microcontroller, and transistor circuitry for the LED. Then nearby have something that holds the motor and laser pointer. After some measurements and building in Fusion 360, I came up with something I was happy with. The only thing that I'd change about the build is the size of the hole the laser is supposed to enter. It was slightly too small and therefore the laser couldn't go through all the way. Besides that, it is worth mentioning that the hole below the mirror is for the potential hull sensor as an end effector and mentioned within the "More Comments/Improvements/Future Plans:" section. Finally, to get the breadboard into the slot the little connector things on the side had to be cut off, but the fit was perfect after that.
